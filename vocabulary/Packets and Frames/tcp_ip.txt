1→TCP/IP Model|A 4-layer networking model (Application, Transport, Internet, Network Interface); a summarised version of the OSI model; connection-based and uses encapsulation.|networking model
2→Connection-based|A defining feature of TCP; a connection must be established between client and server before any data is sent.|TCP concept
3→Three-Way Handshake|The TCP process of establishing a connection: SYN → SYN/ACK → ACK before data transfer begins.|TCP concept
4→https://sterlingalston.github.io/tryhackme/packets%20and%20frames/.attachments/7c5d0e0f92263ede1b06c8aeb0487bd3cf342c19.png|TCP Three-Way Handshake — identify and describe each step shown in this diagram|TCP concept
5→TCP Source Port|A TCP header field containing the randomly-chosen port number opened by the sender (range 0-65535).|TCP header
6→TCP Destination Port|A TCP header field containing the fixed port number of the application or service on the receiving host (e.g. port 80 for a webserver); not chosen at random.|TCP header
7→TCP Sequence Number|A random number assigned to the first piece of transmitted data; used to reconstruct data in the correct order.|TCP header
8→TCP Acknowledgement Number|The next expected sequence number (previous sequence number + 1); confirms receipt of data.|TCP header
9→TCP Checksum|A mathematical calculation stored in the TCP header; if the receiving device's result differs, the data is considered corrupt.|TCP header
10→TCP Flag|A TCP header field that determines how a packet is handled during the handshake process (e.g. SYN, ACK, FIN, RST).|TCP header
11→SYN|The initial TCP handshake packet sent by a client to initiate a connection and synchronise sequence numbers.|TCP handshake
12→SYN/ACK|The second TCP handshake packet sent by the server to acknowledge the client's SYN and provide its own Initial Sequence Number.|TCP handshake
13→ACK|An acknowledgement TCP packet used by either device to confirm that a series of messages/packets have been successfully received.|TCP handshake
14→DATA|A TCP message type used to transmit actual data (bytes of a file) once a three-way handshake connection has been established.|TCP handshake
15→FIN|A TCP packet used to cleanly close an established connection after data transfer is complete.|TCP handshake
16→RST|A TCP packet that abruptly terminates all communication; used as a last resort when a problem occurs (e.g. service not working, low resources).|TCP handshake
17→ISN|Initial Sequence Number; the random number assigned to the first data packet in a TCP connection; each subsequent packet increments by 1.|TCP concept

19→UDP|User Datagram Protocol; a connectionless protocol that sends data without guaranteeing delivery; faster but less reliable than TCP.|protocol
20→UDP Stateless|UDP does not send acknowledgements; there is no guarantee that data is received by the other device.|UDP concept
21→UDP TTL|Time to Live field in a UDP packet header; sets an expiry timer to prevent packets clogging the network.|UDP header
22→UDP Source Port|The randomly-chosen port opened by the sender to send the UDP packet (range 0-65535).|UDP header
23→UDP Destination Port|The fixed port number of the application or service on the receiving host; not chosen at random.|UDP header
24→https://sterlingalston.github.io/tryhackme/packets%20and%20frames/.attachments/2bda37b290faad32c0eaf66c60158d35fc42dc87.png|UDP communication — what does this diagram show about how UDP sends data?|UDP concept
25→UDP Source Address|The IP address of the device sending the UDP packet, stored in the UDP header so data knows where to return to.|UDP header
26→UDP Destination Address|The IP address of the device the UDP packet is being sent to, stored in the UDP header so data knows where to travel next.|UDP header
27→UDP Data|The UDP header field where the actual data bytes being transmitted are stored.|UDP header
